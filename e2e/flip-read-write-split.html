FLIP read/write split: one reflow vs. N

Both passes do identical work — read every row's position, then animate each by its delta. Only the order differs:

  • Interleaved — read a row, animate it, next row (read, write, read, write…).
  • Batched — read every row, then animate every row.

li.animate() dirties the render tree, so the next getBoundingClientRect() can't answer until the browser reflows. Interleaved pays that reflow once per row; batched pays it once. The wall-clock gap below is the tell — the interleaved loop is O(N²).
