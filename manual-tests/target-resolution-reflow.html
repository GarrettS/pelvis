Target resolution: elementFromPoint vs frozen numbers

The sortable-list drag resolves the pointer to a drop slot once per pointermove. Two ways to do that resolve:

  • elementFromPoint — document.elementFromPoint(x, y), then getBoundingClientRect() on the result for its midpoint. A DOM read.
  • frozen numbers — midpoints measured once, then each resolve is array.find(({midpointY}) => pageY <= midpointY). No DOM read.

This page runs each resolve times, timed twice: with page layout clean, and with layout dirtied before each resolve. Clean isolates the hit-test cost; dirty adds a forced reflow per resolve. The numbers scan reads nothing either way.

Builds a 2000-item list and runs 5 trials per cell.

Exact reflow counts (Chromium's LayoutCount) are readable only over DevTools/CDP, not from in-page JavaScript. This page measures the wall-clock consequence instead.
